Implementation Spec: Variability Policy, Catalogue–Rule Consistency, and Hyrox Simulation Repo: bubble-workout-engine Branch target: new feature branch from feature/pipeline-multitype-refactor Date: 2026-03-29 Status: Spec only — no code changes 1. Executive Summary Feature 1 — Variability Policy Add a variability_policy field (none | med | high) to slot and block-default definitions. A new VariabilityState object, instantiated per-program, tracks sticky exercise choices keyed by an explicit slot_family string (falling back to sw2 if unset). none always returns the previously chosen exercise for a family and bypasses week-dedup. med is sticky within a block but adds prior choices to a soft-avoid set across blocks. high is current behavior unchanged. This is a general mechanism — not Hyrox-specific — and is wired through fillSlot → pickWithFallback via an augmented builderState. Feature 2 — Catalogue–Rule Consistency The concrete drift risk today is that exercise fields (sw2, movement_class, preferred_in_json) can change without triggering any rep-rule coverage check. The v1 solution is an Admin Health Report (new page in the admin panel) that runs at query time against live DB state and surfaces: uncovered exercises (fall through to global fallback only), orphaned rule dimensions (no exercises match), pref-tag mismatches, and per-slot coverage gaps. A secondary CI guard is a seeded SQL assertion test. No schema changes required in v1. Feature 3 — Hyrox Simulation The DB already has hyrox_role and hyrox_station_index on the exercise catalogue, and a hyrox_default_v1 config skeleton. V1 adds a simulation day-type to the hyrox config with is_ordered_simulation: true, ordered station slots keyed by hyrox_station_index, explicit slot_family + variability_policy: "none" per slot, and station_fallback_chain arrays that define graceful equipment degradation. The selector gains a simulationDegradation field in its per-item output (exact / family / fallback / unresolvable) surfaced in admin preview. 2. Current-State Assessment 2.1 Exercise Selection (Step 01) File: api/engine/steps/01_buildProgramFromDefinition.js State is reset per day (not per program or per block): usedIdsWeek — IDs used anywhere this week across days usedSw2Today — sw2 groups used in this day usedCanonicalNamesToday — canonical names used today (prevents equipment variants of the same movement) usedRegionsToday — target regions trained today There is no program-level or block-level sticky state. There is no concept of "this family must always resolve to exercise X." The scoring in pickBest is deterministic given the same allowedSet order, but allowedSet is not guaranteed ordered identically across runs; and usedIdsWeek exclusions mean the top-scored exercise for a slot on day 2 differs from day 1 if it was already used on day 1. For Hyrox running slots today, this means "Run Intervals (Outdoor/Treadmill)" is top-scored on day 1 slot A:run, excluded by usedIdsWeek on day 2, so "Outdoor Run" wins on day 2. This is the exact fragmentation problem Feature 1 must solve. pickWithFallback (in api/engine/exerciseSelector.js) chains ~9 tiers of relaxation. The tiers exist to find a match if the preferred exercise is excluded by usedSet (week dedup). There is no mechanism to intentionally force a specific exercise when it is already in usedSet. 2.2 Selector Strategies File: api/engine/selectorStrategies.js One strategy today (best_match_by_movement). It accepts a state parameter that currently carries usedSet, usedRegionsSet, usedSw2Set, usedCanonicalNamesSet, condState, condThresholds, stats. Adding variabilityState here follows the established pattern. 2.3 Slot / Config Shape File: migrations/R__seed_program_generation_config.sql Each slot def today: { "slot": "A:squat", "sw2": "squat_compound", "requirePref": "strength_main", "pref_mode": "strict", "preferLoadable": true, "variants": [ { "when": { "equipment_profile": "full" }, "sw2": "squat_compound" }, { "when": { "equipment_profile": "minimal" }, "swAny": ["squat_pattern"] } ], "fill_fallback_slot": "A:squat" } No variability_policy, no slot_family. 2.4 Rep Rules (Step 04) File: api/engine/steps/04_applyRepRules.js Rule matching context per item is built from the exercise catalogue fields at rule-application time: mp → context.movement_pattern sw2 → context.swap_group_id_2 mc → context.movement_class eq → context.equipment_slug (derived from first equipment item) tr → context.target_regions Plus segment context (purpose, segment_type) and day context (day_type). program_type is the only mandatory field. This means every exercise always gets a rule — but getting a low-priority fallback rule is often a silent degradation. 2.5 Drift Risks Today Add a new exercise with sw2: "squat_compound" but forget to give it preferred_in_json: ["strength_main"] → selector treats it like any other sw2 match; pref-gated slots open to it unexpectedly. Candidate Designs Feature 1 — Variability Policy Option A: slot_family string + variability_policy per slot (Recommended) Add explicit fields to each slot definition: { "slot": "A:run", "slot_family": "hyrox_run", "variability_policy": "none", "sw": "locomotion" } VariabilityState object (new, created per program run): programSticky: Map<slotFamily, exerciseId> blockSticky: Map<"blockLetter:slotFamily", exerciseId> blockHistory: Map<slotFamily, Set<exerciseId>> Selector consults variabilityState before calling pickWithFallback: none: if programSticky.has(family) → return that exercise (bypass scoring and week-dedup entirely) med: if blockSticky.has(blockKey+family) → return sticky; else pass blockHistory.get(family) as a soft-avoid set and call normal pickWithFallback high: no-op, current behavior Pros: Explicit, general, no magic. Works for any program type. Easy to audit in config. slot_family can be set on non-Hyrox slots too (e.g., all "shoulder press" slots family = "vertical_push_family" for med policy in hypertrophy). Cons: More fields to populate in config. Block letter identity ("A", "B") needs to be passed through to selector state. Option B: Derive family from sw2 automatically Use the slot's resolved sw2 value as the implicit family key. No new field needed. Pros: Zero config changes to existing slots. Cons: sw2 is a movement grouping, not a "pick the same exercise" identity. Multiple slots can share the same sw2 but reasonably need different exercises. Also sw2 is often null for conditioning/locomotion slots. Would need sw as a fallback key, but sw is even less unique. Brittle, especially for Hyrox where you want to distinguish between the "run before SkiErg" and the "run before SledPush." Option C: Per-program-type variability override (e.g., hyrox always none) Apply variability policy at program type level, not slot level. Cons: Too coarse. Hypertrophy programs want med on some families and high on most. Strength programs might want none on main compound lifts within a mesocycle. This doesn't model the needed granularity. Recommendation: Option A. Feature 2 — Catalogue–Rule Consistency Option A: Admin Health Report page (Recommended for v1) A new admin page (/admin/health.html backed by /admin/health endpoint) that queries live DB state and computes: Rule coverage matrix: for each active exercise × each program type → which rule it matches and at what priority tier (specific / fallback only) Orphaned rule dimensions: rules with swap_group_id_2 / movement_pattern values that no active exercise has Orphaned pref tags: pref tags referenced in requirePref slots (from config JSON) that no active exercise carries Uncovered exercises: exercises that match only the global fallback rule (priority 1) for a given program type Slot coverage gaps: for each slot in each config × each equipment preset → number of eligible exercises Pros: Zero new infrastructure. Catches drift at human-readable level. Admin can run it on demand before/after catalogue edits. No CI complexity. Cons: Passive — only catches drift when someone looks. Does not block bad migrations from being committed. Option B: SQL assertion test in CI Add a test file (api/test/catalogueRuleHealth.test.js) that queries the DB, runs the same checks as Option A, and asserts no critical mismatches. Pros: Automated. Catches drift before deploy. Cons: Requires a running DB in CI (already the case per architecture — tests use real DB). Adds maintenance burden if new valid orphaned dimensions are introduced deliberately. Option C: DB foreign-key-style constraint Add a constraint_check function or trigger that validates swap_group_id_2 values in program_rep_rule exist in at least one active exercise. Cons: Overly rigid. Orphaned rules are sometimes intentional (placeholder for exercises not yet catalogued). Also, trigger-based constraints are hard to reverse. Not recommended. Option D: Derived rule matching from catalogue Eliminate swap_group_id_2 from program_rep_rule entirely and derive it from the matched exercise at rule-application time. Cons: This collapses too much into the exercise. Rep rules deliberately allow movement-pattern-based overrides regardless of the specific exercise chosen. This redesign is a large v2 project, not a v1 fix. Recommendation: Option A as v1, Option B as v2. Recommended Designs Feature 1: Variability Policy via slot_family + VariabilityState Variability Policy is a slot-level field with an optional block-level default in config. The effective policy for a slot is: slot's own variability_policy → block default → "high" (current behavior). slot_family is an explicit string key. When absent, the selector treats it as non-sticky (high behavior regardless of variability_policy). This means existing slots without slot_family are unaffected by this change. VariabilityState is instantiated once per buildProgramFromDefinition call and passed through to fillSlot via builderState. It is never serialized to the DB; it is ephemeral to one generation run. none semantics: On first selection for a family: score normally, record winner in programSticky On subsequent selections for same family: return programSticky[family] directly, bypassing usedSet check (same as pickSeedExerciseForSlot already does for day-0 seeding) Result: same exercise every time across the entire program med semantics: On selection for a family within a block: if blockSticky[blockKey:family] is set, return it directly If not yet set for this block: call pickWithFallback but add blockHistory[family] as additional entries in avoidCnSet (soft avoid, not hard exclude from week-dedup) Record winner in both blockSticky and blockHistory Result: same exercise within a block; prefers novelty across blocks but accepts repeats when no viable alternative exists Interaction with existing logic: none exercises are excluded from usedIdsWeek push (or explicitly re-included on retrieval). The week-dedup must not block the sticky exercise. med uses existing avoidCnSet mechanism — canonical names of prior-block winners are soft-avoided via the same scoring penalty (-0.3 per shared canonical name in avoidCnSet) high is fully unchanged Feature 2: Admin Health Report A new Express route (GET /admin/health/report) returns a JSON payload. A new admin page (/admin/health.html) renders it in table form. The report runs on demand (no caching in v1). Report sections: rule_coverage — per active exercise per program type: { exercise_id, name, program_type, matched_rule_id, matched_rule_priority, is_fallback_only } orphaned_rules — rules whose swap_group_id_2, movement_pattern, or movement_class values appear in no active exercise orphaned_prefs — requirePref values referenced in any config slot that appear in no active exercise's preferred_in_json uncovered_exercises — active exercises that match only the global fallback (priority ≤ 1) for at least one program type they could plausibly appear in slot_coverage — for each config slot × equipment preset: count of eligible exercises (< 2 = warning, 0 = critical) Feature 3: Hyrox Simulation Day The simulation day is added as a new template inside hyrox_default_v1.builder.day_templates. The prescribed order (run → station 1 → run → station 2 → ... → station 8 → run) is modeled as an array of station slots with hyrox_station_index and run_leg_index. Each week in the hyrox program can include one simulation day alongside training days. Whether a simulation day appears at all in a given week is controlled by week phase: PEAK weeks include simulation; BUILD and BASELINE weeks use training-only templates. Equipment fallback is modeled as station_fallback_chain: an ordered array of selector specs tried in sequence. The selector tries each entry until one resolves. The first entry is the exact race-spec; subsequent entries are progressively more permissive. Each entry can carry equipment_required: ["ski_erg"] — if the user's equipment profile lacks this slug, skip this entry. Fitness level scaling uses the existing variants mechanism (already supported in slot resolution). 5. Selector Logic Changes 6.1 VariabilityState — new module File: api/engine/variabilityState.js (new) export function createVariabilityState() { return { programSticky: new Map(), // familyKey → exerciseId blockSticky: new Map(), // "blockLetter:familyKey" → exerciseId blockHistory: new Map(), // familyKey → Set<exerciseId> }; } export function resolveStickyExercise(state, familyKey, blockLetter, policy, byId) { if (!familyKey || policy === "high") return null; if (policy === "none") { return state.programSticky.get(familyKey) ?? null; } if (policy === "med") { const bKey = ${blockLetter}:${familyKey}; return state.blockSticky.get(bKey) ?? null; } return null; } export function recordStickyChoice(state, familyKey, blockLetter, policy, exerciseId) { if (!familyKey || !exerciseId || policy === "high") return; if (policy === "none") { if (!state.programSticky.has(familyKey)) { state.programSticky.set(familyKey, exerciseId); } } else if (policy === "med") { const bKey = ${blockLetter}:${familyKey}; if (!state.blockSticky.has(bKey)) { state.blockSticky.set(bKey, exerciseId); } if (!state.blockHistory.has(familyKey)) state.blockHistory.set(familyKey, new Set()); state.blockHistory.get(familyKey).add(exerciseId); } } export function getMedAvoidSet(state, familyKey) { return state.blockHistory.get(familyKey) ?? new Set(); } 6.2 Changes to buildProgramFromDefinition Create variabilityState at top of function, before the day loop. 9. Implementation Plan

Phase 1 — Variability Policy (v1)
Goal:
Introduce a general, config-driven variability mechanism that supports:

* `none`: sticky across the full program
* `med`: sticky within block, prefers variation across later blocks
* `high`: current behaviour unchanged

Scope:

* selector state
* slot/config schema
* config editor support
* unit tests
* no DB schema migration required beyond seed/config changes

Implementation steps:

1. Add config fields to the PGC parser / validation path

   * Extend the config shape to accept:

     * `slot_family?: string`
     * `variability_policy?: "none" | "med" | "high"`
   * Extend builder-level block defaults:

     * `block_variability_defaults?: Record<string, "none" | "med" | "high">`
   * Ensure absence of these fields preserves existing behaviour.

2. Add `VariabilityState` module

   * Create `api/engine/variabilityState.js`
   * Include:

     * `createVariabilityState()`
     * `resolveStickyExercise(...)`
     * `recordStickyChoice(...)`
     * `getMedAvoidSet(...)`
   * Keep this module pure and ephemeral to one generation run.

3. Thread variability state through Step 01

   * In `01_buildProgramFromDefinition.js`:

     * instantiate `variabilityState` once per program build
     * pass it through `builderState`
     * resolve effective policy per slot:

       * slot `variability_policy`
       * else block default
       * else `"high"`
   * Resolve effective family key:

     * `slot_family` if present
     * else `null`

4. Implement `none`

   * Before normal fill/selection:

     * if policy is `none` and sticky exercise already exists for family, reuse it directly
   * On first normal selection:

     * store winner in `programSticky`
   * Bypass week-dedup for reused sticky winner
   * Preserve all existing metadata on emitted block entry

5. Implement `med`

   * Reuse same exercise within a block if already selected for that block/family
   * If selecting first time in a block:

     * call normal selector
     * augment soft-avoid inputs with prior block-history for that family
   * Do not force a lower-quality match if only one viable option exists
   * Record chosen winner in:

     * `blockSticky`
     * `blockHistory`

6. Leave `high` unchanged

   * This should route entirely through current logic.

7. Add debug visibility

   * Add selector debug events for:

     * effective policy
     * family key
     * sticky hit / miss
     * med soft-avoid candidates
   * Surface this in program preview/admin debug payload if already supported.

8. Update config editor

   * Add per-slot controls:

     * Variability Policy dropdown
     * Slot Family text field
   * Add optional block default controls if editor already supports block-level settings
   * Do not require these fields for non-Hyrox programs

9. Seed/config updates for initial usage

   * Update Hyrox run-family slots to use:

     * `variability_policy: "none"` for tightly fixed cases
     * `slot_family: "hyrox_run"` or similarly explicit family keys
   * For conditioning cases where block-level machine consistency matters:

     * use `med` with suitable family keys

12. Risks / Tradeoffs

Risk: `slot_family` is configured inconsistently
Impact:

* sticky behaviour becomes confusing or too broad
  Mitigation:
* admin preview should show family keys
* health report can later flag suspicious family reuse patterns

Risk: `med` produces too much or too little variation
Impact:

* athlete experience feels odd or repetitive
  Mitigation:
* keep v1 soft-avoid only
* tune later with scoring/debug data instead of hard exclusions initially

Risk: simulation config becomes large and tedious to maintain
Impact:

* higher seed complexity
  Mitigation:
* keep it fully config-driven
* use explicit fallback chains only where needed
* consider editor support later if simulation authoring becomes frequent

Risk: health report duplicates core matching logic
Impact:

* drift between runtime and diagnostics
  Mitigation:
* extract shared rule-matching helpers where possible

Risk: ordered simulation path diverges from standard day generation path
Impact:

* extra complexity in Step 01
  Mitigation:
* keep the ordered path narrow and only activated by `is_ordered_simulation`

Risk: `none` bypasses dedup too aggressively
Impact:

* too much repetition in the wrong places
  Mitigation:
* only active when explicitly configured with a family key
* default behaviour remains unchanged
